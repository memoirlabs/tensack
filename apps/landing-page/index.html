Backend Code

Each crate owns one boundary. Keep storage details in the store, byte rules in the format crate, and user-facing calls in the composed runtime API.

packages/tensack-core

Domain model

Schemas, table definitions, primitive field types, records, values, and validation errors.

packages/tensack-format

Durable format

Encodes and decodes `.ten` preambles, put/delete rows, row pointers, source hashes, and `.tenb` cache data.

packages/tensack-store

Local storage

Creates directories, writes reverse-sorted chunks, scans row files, and rebuilds lookup caches.

packages/tensack

Runtime API

The public `TensackDatabase` API for init, `get` selectors, `write` changes, and cache rebuilds.

packages/tensack-cli

CLI behavior

Command parsing and execution. Current shipped commands are intentionally tiny: help and version.

packages/tensack-schema-compiler

Schema compiler

Parses `schema!` snippets, validates names/lookups/duplicates, and emits raw Rust row/table helpers.

Write Path

A write stays simple: validate the record, encode one row, write one deterministic chunk, then rebuild the generated cache.

1

Validate schema and record

The runtime finds the table and checks fields before storage sees bytes.

2

Assign counters

`next_tx` gives the row transaction id. `next_chunk` gives the chunk filename.

3

Write the chunk

The store creates paths like `tables/messages/zz/zzz.ten` using reverse lowercase base-36.

4

Refresh generated state

`.tenb` is rebuilt from canonical `.ten` chunks and `tensack.toml` records the next counters.

counter 0 -> zz/zzz.ten
counter 1 -> zz/zzy.ten
counter 2 -> zz/zzx.ten

generation = counter / 46656
local      = counter % 46656

folder = reverse_base36(generation, 2)
file   = reverse_base36(local, 3)

Public API

The Rust API is the main usable surface right now. The CLI is present, but intentionally only exposes help/version until the command contract is real. 

let db = TensackDatabase::open_local_with_schema(
    root,
    "chat",
    schema,
);

db.init()?;
db.write(change::add(row))?;
db.write(change::set(replacement))?;
db.write(change::remove_id("messages", "m1"))?;

let row = db.get(selector::id("messages", "m1"))?;
let rows = db.get(selector::many(
    "messages",
    "conversation_id",
    "cv1",
))?;

Target generated API

The public direction is `get` / `watch` / `write` with schema-generated values.

  • `db.get(messages::by::id("m1"))`
  • `db.get(messages::by::conversation_id("cv1"))`
  • `db.get(messages::all().limit(100))`
  • `db.get(messages::count())`
  • `db.write(messages::add(row))`
  • `db.write(messages::edit(target, patch))`
  • `db.write(messages::remove(target))`
  • `db.watch(selector, callback)` is planned for subscriptions.

Current CLI

The command surface is deliberately narrow while storage and schema behavior settles.

  • `tensack --version` prints the version.
  • `tensack help` prints usage.
  • No interactive shell is shipped.
  • No inspect/repair/migration command is claimed yet.

Status

This is the honest line between the backend that exists and the design work that is still only planned.

Implemented now

  • Typed schema primitives and record validation.
  • Readable `.ten` row encoding for put/delete operations.
  • Reverse-sorted 2-folder plus 3-file chunk naming.
  • `tensack.toml` metadata with `next_tx` and `next_chunk`.
  • Generated `.tenb` lookup cache rebuilds.
  • Rust API for `get` selectors, `write` changes, and lookup-backed state access.

Planned, not shipped

  • Admin database viewer.
  • Richer CLI commands beyond help/version.
  • Generated `db.<table>` API and plan envelope executor.
  • Compaction, repair, and inspection commands.
  • Binary-packed `.tenb` layout.
  • `.tenx` full-text search.
  • Generated registry surface beyond the current compiler proof.
